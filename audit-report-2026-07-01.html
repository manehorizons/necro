Executive summary

necro's core is in unusually good shape. The codebase typechecks clean under maximal strictness with zero escape hatches, the 385-test suite passes in under five seconds with acceptance-criteria-traceable titles, the npm release pipeline is guarded and provenance-signed, and the accuracy story is backed by a real, SHA-pinned eval harness rather than marketing copy. The engineering discipline here is the product's moat.

The problems are almost all at the boundary between the code and its users. Everything shipped since June 11 — explain, verify-removal, explain --narrate, the Next.js plugin, monorepo support — is invisible: not on npm, not in the README, not in the CHANGELOG, and the docs site still calls explain "planned." Meanwhile no CI workflow runs the (fast, free) test suite on push or PR, so the safety net only exists at release time, and the live accuracy gates never fire anywhere.

Two real bugs were confirmed by exercising the CLI live: verify-removal always exits 0 even when a removal breaks the build (defeating its headline CI use case), and --checks splits naively on commas, shredding any check command containing one — and then misreports the mangled command's failure as "removal breaks the build."

The highest-leverage week of work available: add a CI test workflow, fix the two bugs, sync the docs to HEAD, and cut v1.2.0. That single sequence converts about six weeks of already-finished feature work into something users can actually install and trust.

Confirmed bugs

Both reproduced live during the audit against a planted fixture repo — these are observations, not code-reading speculation.

verify-removal always exits 0 — even when removal breaks the build

src/cli.ts:162–178 · severity: high

The command prints the right verdict but never sets process.exitCode, unlike explain. Its headline use case — CI-gating a deletion PR — silently cannot work; the only workaround is parsing --json yourself.

✗ used — removal breaks the build
$ echo $?
0   ← should be non-zero

--checks splits on every comma, shredding real commands

src/cli.ts:166 · severity: medium (corrupts verdicts)

opts.checks.split(",") breaks any check command containing a comma — and the mangled fragment's shell error is then reported as "removal breaks the build," a false verdict. Accept a repeatable flag or a JSON array instead.

--checks "node -e 'f(1,2)'"
/bin/sh: Syntax error: Unterminated quoted string
✗ used — removal breaks the build  ← wrong conclusion

MCP server & LLM integration

score 82

Four read-only stdio tools, verified live (handshake, tools/list, clean shutdown). The LLM layer is notably current: claude-opus-4-8 default, adaptive thinking, JSON-schema structured outputs — no fragile text parsing anywhere.

Strong: MCP tools are thin adapters over the same engine the CLI uses (a golden-equality test forbids a fork); malformed model responses degrade to unsure instead of throwing; spend caps exist (maxFindings, --limit); code goes to the API only via explicitly opt-in commands, and that's documented.

  • MedNo progress notifications or duration warnings on long-running tools — necro_verify_removal runs a full typecheck+test cycle per symbol; MCP clients with default 60s timeouts will abort it mid-run.
  • MedMCP tools load config from the server's cwd, not the scan target — scanning a sibling project silently ignores that project's necro.config.json.
  • MedNo token/cost visibility — response.usage is never read, and maxFindings is uncapped by default, so a large-repo triage is unbounded spend.
  • LowOnly explicit API keys are accepted — ANTHROPIC_AUTH_TOKEN and CLI-login profiles that the zero-arg SDK client resolves automatically are rejected.
  • LowNo typed 429/connection error handling or partial-batch recovery (a rate-limited 200-finding run dies mid-batch); the "prompt-cached" comments are aspirational — no cache_control is ever set.
