# 20-20 — ship & adopt: publish @manehorizons/necro + CI/PR citizen

> Derived from milestone `mil-grp-ship-adopt` (recs rec-20260610-004 publish,
> rec-20260610-005 CI/PR citizen). One milestone, two acceptance criteria.

## Objective

Make necro adoptable in practice: publish `@manehorizons/necro` to npm so an agent
can install and run the phase-19 MCP server with no clone, and turn necro into a
CI/PR citizen with SARIF output, a packaged GitHub Action, and `--fail-on`
severity gating.

## Acceptance Criteria

### AC-1: Publish @manehorizons/necro to npm + working agent-install story
Given `@manehorizons/necro` is unpublished (`npm view @manehorizons/necro version` → 404),
`package.json` version is `0.0.0`, and the only way to run `necro mcp` is
`git clone && npm run build`
When the version is bumped off `0.0.0` and the scoped package is published to npm
(bin `necro` → `dist/cli.js`, `files:["dist"]`, prepublish build), a tag → build →
publish release workflow is added, and the README/site MCP config is switched to
the `npx` form
Then `npm view @manehorizons/necro version` returns the published version, and
`npx -y @manehorizons/necro mcp` starts the stdio MCP server from the published
package alone (no local clone), so the documented agent MCP config
(`{command:"npx", args:["-y","@manehorizons/necro","mcp"]}`) is real

### AC-2: CI/PR citizen — SARIF output + GitHub Action + `--fail-on` gating
Given necro emits human and JSON reports but has no SARIF output, no packaged
GitHub Action, and no severity-based exit gating
When a SARIF reporter (`src/report/sarif.ts`) is added and wired to a CLI output
flag, a composite GitHub Action (`.github/actions/necro/action.yml`) wraps the
scan, and `--fail-on <severity>` gates the process exit code
Then running necro with SARIF output produces a schema-valid SARIF file that
GitHub code-scanning accepts on upload, the Action runs necro against a PR and
surfaces findings, and `--fail-on high` exits non-zero iff a finding at or above
the threshold exists (zero otherwise)

## Constraints

- Scope is ship + adopt only. Out of scope: measured-accuracy benchmark
  (rec-006) and false-positive-reduction plugins (rec-008) — reserved for a
  separate accuracy milestone; deepening the agent wedge / `necro explain`
  (rec-007); non-GitHub CI providers (GitLab/CircleCI/etc.) — GitHub Action +
  SARIF only.
- Publish scoped-only as `@manehorizons/necro` (MIT, `private:false`) unless the
  publish surfaces a hard name conflict; no unscoped alias by default.
- Touches documentation surfaces (README + Astro site) — keep docs in lockstep
  with the shipped surface to avoid spec/doc drift. The Astro docs site needs
  Node ≥ 22 to build.

## Open Questions

- **Phase split:** both ACs edit `src/cli.ts` (shared-file coupling flagged by
  the milestone pre-mortem). Decide at DRAFT time whether to sequence as one
  phase or split publish (AC-1) and CI-citizen (AC-2) into two, and in what
  order, to avoid colliding edits.
- **Publish credentials:** does an npm token / 2FA path exist for the
  `@manehorizons` scope? A missing token or 2FA gate blocks AC-1 entirely.
- **SARIF severity mapping:** how do necro finding severities map to SARIF
  levels and to the `--fail-on` threshold, so gating is neither noisy nor silent?
- **Action validation:** how is `action.yml` exercised against a real PR (not
  just unit tests) before it is relied on?
