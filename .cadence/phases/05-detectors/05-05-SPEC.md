# 05-05 — Syntactic detectors (tree-sitter)

> Source of truth: [docs/necro-design-spec.md](../../../docs/necro-design-spec.md).
> This slice = build-order step 8 (§8): syntactic detectors (nesting,
> cyclomatic, cognitive, god-function) via tree-sitter. Introduces the
> language-agnostic **syntactic IR** (§3.1) fed by tree-sitter; detectors run on
> the IR, never on TS nodes (core invariant §3). Duplication (step 9), CRAP
> score (step 10), and churn are later phases.

## Objective

Add a second analysis axis to necro: parse functions into a language-agnostic
syntactic IR via tree-sitter and flag over-complex code — deep nesting,
high cyclomatic/cognitive complexity, and god-functions — surfaced as a distinct
`complexity` section in `necro scan`, deterministic and free.

## Acceptance Criteria

### AC-1: Language-agnostic syntactic IR via tree-sitter
Given a TypeScript source file with functions
When it is parsed
Then each function is lowered to an IR unit — `{ name, file, line, loc, params, controlNodes: { kind, depth }[] }` — produced by tree-sitter (decision #6), and the lowering (tree-sitter node kinds → IR) is the **only** code that names TS/tree-sitter constructs; the detectors that follow read the IR alone.

### AC-2: Nesting-depth detector
Given a function whose maximum block-nesting depth exceeds the threshold (default 3)
When detectors run
Then a `nesting` finding reports the function, its measured depth, and the threshold; a function at depth ≤ 3 is not flagged.

### AC-3: Cyclomatic-complexity detector
Given a function whose cyclomatic complexity (1 + branch/loop/boolean-operator count) exceeds the threshold (default 10)
When detectors run
Then a `cyclomatic` finding reports the measured value and threshold; a function at ≤ 10 is not flagged.

### AC-4: Cognitive-complexity detector (nesting-weighted)
Given two functions with the same number of branches — one flat, one deeply nested
When the cognitive detector scores them (Sonar model: increment per control structure, extra penalty per nesting level)
Then the nested function scores strictly higher than the flat one, and a function above the threshold (default 15) is flagged `cognitive` — distinguishing human-pain from raw branch count.

### AC-5: God-function detector
Given a function exceeding the LOC threshold (default 50) or the parameter-count threshold (default 5)
When detectors run
Then a `god-function` finding reports which limit was exceeded and the measured value; responsibility-cluster (distinct-callee) analysis is explicitly out of scope for this slice.

### AC-6: Multi-axis scan output
Given a project containing both dead code and over-complex functions
When `necro scan` runs
Then it prints the existing dead-code findings, then a distinct, labeled `complexity` section sorted worst-first by severity; `--json` emits a `complexity` array alongside `findings`; a simple, in-threshold function appears in neither section; and `necro fix` is unaffected (it still operates only on dead-code findings).

## Constraints

- **Parser is tree-sitter** (`tree-sitter-typescript`) — locked decision #6 (tree-sitter for syntactic detectors; TS compiler API stays for dead-code). Planned binding: **web-tree-sitter (WASM)** for a portable single esbuild distributable and the polyglot grammar story; lazy-loaded (heavy). Native `node-tree-sitter` is the fallback if WASM↔esbuild asset loading proves unworkable — captured as a draft risk.
- **Core invariant §3:** detectors operate on the syntactic IR only. Language-specific code lives **solely** in the tree-sitter lowering step. A detector that special-cases a language is a leak and must be rejected in review.
- **Detectors + default thresholds** (§4): nesting > 3, cyclomatic > 10, cognitive > 15, god-function LOC > 50 / params > 5. Thresholds are overridable via `necro.config.json` (a `complexity` block); defaults apply when unset.
- **Surfacing:** complexity findings appear in `necro scan` as a separate section + a `--json` `complexity` array. The dead-code output and tier logic are unchanged; `ScanResult` gains a `complexity` field that `necro fix` ignores.
- **Out of scope (later phases):** god-function responsibility-cluster analysis, duplication (step 9), CRAP score + churn (step 10), and any LLM use (locked #3).
- **Bundling:** `npm run build` (esbuild) must still emit a working single CLI; the tree-sitter grammar asset must be locatable at runtime from `dist/`.
- Detectors are **report-only** — no auto-fix for complexity in this slice (LLM refactors are step 13).

## Decisions (resolved open questions)

- **Surfacing:** integrated into `necro scan` as its own section (multi-axis), not a separate command — matches the single-scan design vision.
- **Detector scope:** all four step-8 detectors; god-function is the simple LOC + param-count heuristic, responsibility-clustering deferred.
- **Parser binding:** web-tree-sitter (WASM) planned; native fallback noted as a draft risk.
- **Thresholds:** design defaults, overridable via a `complexity` config block.
